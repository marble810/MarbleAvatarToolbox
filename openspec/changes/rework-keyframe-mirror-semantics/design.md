## Context

当前 KeyframeMirror 同时承载了两类不同语义：

- Mirror Keyframes：用户期望对当前选中姿态做自体空间镜像。
- Key Symmetry：基于左右骨配对，把一侧姿态镜像后复制到另一侧。

现状中这两类行为共用了同一套“按曲线名判断是否取反”的处理路径，导致 Mirror Keyframes 实际表现为对单条标量曲线做条件取反，而不是对完整姿态样本做空间反射。对于旋转曲线、部分通道选中、四元数曲线和插值信息，这种做法都会产生错误或不稳定结果。

同时，当前实现缺少镜像平面的参考空间定义，只有 X/Y/Z 三个枚举，没有说明镜像平面经过哪里、使用谁的坐标轴，也没有说明当用户只选中某个 Transform 的部分通道时应如何聚合处理。

## Goals / Non-Goals

**Goals:**
- 将 Mirror Keyframes 明确定义为基于参考空间镜像平面的姿态反射操作。
- 明确 X/Y/Z 与参考空间中 YZ/XZ/XY 平面的对应关系。
- 明确选区聚合规则：当同一 Transform 在同一时间点有任一相关通道被选中时，以完整 Transform 样本为单位执行镜像。
- 将 Mirror Keyframes 与 Key Symmetry 在内部处理上拆分为“共享数学内核、不同目标写回策略”的两个模式。
- 为位置、欧拉旋转、四元数旋转和关键帧写回建立明确边界，避免继续依赖字符串包含和标量取反。
- 将 BoneKeyMirror.cs 作为废弃旧脚本纳入清理范围，避免遗留重复入口和重复类定义。

**Non-Goals:**
- 不在本次设计中扩展为任意属性的通用几何镜像框架。
- 不改变与 Transform 无关的非姿态曲线的语义。
- 不尝试在本次变更中支持用户自定义任意旋转镜像平面。
- 不把 KeyframeMirror 扩展为完整的动画重定向或批量重映射系统。

## Decisions

### 1. Mirror Keyframes 的参考空间固定为 Animation Window 活动根对象的本地空间
- 决策：Mirror Keyframes 使用 Animation Window 当前活动根对象作为参考根，镜像平面始终经过该根对象原点，并使用其本地坐标轴定义 X/Y/Z。
- 原因：这与用户在动画编辑时对“角色中心线”的直觉最接近，也比“每个骨骼各自父空间镜像”更符合姿态级空间对称的预期。
- 备选方案：
  - 使用每个 Transform 的父空间。放弃原因：会把空间镜像退化成局部通道翻转，不符合整体姿态镜像语义。
  - 新增用户可选任意参考 Transform。放弃原因：首版先钉死语义，避免 UI 和数据模型继续发散。

### 2. 选区按 Transform + 时间点聚合，任一通道入选即镜像完整样本
- 决策：当用户选中某个 Transform 在时间点 t 的任一位置或旋转相关通道时，系统先重建该 Transform 在 t 的完整姿态样本，再执行镜像并写回对应的完整曲线组。
- 原因：用户真正编辑的是“姿态”，不是孤立标量；只改单个通道会产生半截镜像，结果不可预期。
- 备选方案：
  - 仅处理被选中的标量 key。放弃原因：这正是当前错误来源之一。
  - 强制要求用户总是选齐所有通道。放弃原因：违背 Animation Window 的自然操作方式。

### 3. 姿态镜像以内核“空间反射”实现，而不是曲线值取反
- 决策：对每个聚合样本，先在参考空间中恢复姿态，再通过镜像矩阵执行反射；X 对应 YZ 平面，Y 对应 XZ 平面，Z 对应 XY 平面。之后再解回到目标 Transform 的局部姿态并写回曲线。
- 原因：只有空间反射才能稳定表达“镜像/对称”，并统一处理位置与旋转。
- 备选方案：
  - 继续对位置和旋转曲线分开做经验公式取反。放弃原因：对层级姿态与四元数不成立。

### 4. Mirror Keyframes 与 Key Symmetry 共享数学内核，但写回目标不同
- 决策：Mirror Keyframes 把镜像后的样本写回原 Transform；Key Symmetry 则基于左右骨映射，将源样本镜像后写回目标 Transform。
- 原因：两者都需要同一套“采样 -> 反射 -> 解回”的姿态内核，但用户意图不同，不应再混成一个按钮逻辑。
- 备选方案：
  - 完全拆成两套独立处理流程。放弃原因：重复实现会让后续一致性更差。

### 5. 曲线写回必须保留原始表示与编辑器插值语义
- 决策：如果原曲线使用欧拉表示，则按欧拉组写回；如果原曲线使用四元数表示，则按四元数组写回。关键帧写回必须保留可恢复的 tangent mode、broken/constant、weighted mode 与权重信息，而不是只保留数值和 weighted mode。
- 原因：镜像正确不仅是值正确，还必须保持动画曲线在编辑器中的可预期行为。
- 备选方案：
  - 统一转换成欧拉或统一转换成四元数。放弃原因：会破坏现有剪辑的表示和插值预期。

## Risks / Trade-offs

- [风险] 参考根对象解析失败或与用户心理预期不一致 → [缓解] 在实现中显式提示当前参考根，并在无法解析时拒绝执行而不是隐式回退。
- [风险] 四元数与欧拉互转可能造成视觉上等价但数值不同的写回 → [缓解] 只在原表示内写回，避免无必要的跨表示转换。
- [风险] 同一时间点缺少部分通道 key 时，完整样本重建需要依赖曲线采样和默认值 → [缓解] 明确采样优先级，并将其作为 spec 行为写清。
- [风险] 层级解回会比当前逐曲线处理更复杂 → [缓解] 通过“采样/反射/写回”分层设计降低复杂度，并把 Key Symmetry 复用到同一内核。
- [风险] 旧脚本 BoneKeyMirror.cs 与新脚本并存会导致重复类或入口混乱 → [缓解] 将旧脚本删除纳入本 change 的明确任务。

## Migration Plan

- 该变更只影响编辑器工具和相关 OpenSpec 文档，不涉及运行时数据迁移。
- 实现阶段先替换 KeyframeMirror 的镜像内核，再删除 BoneKeyMirror.cs 及其关联资产引用。
- 如重构中发现参考空间定义仍需 UI 暴露，可在同一 change 内追加设计说明，但不改变本次已钉死的默认语义。

## Open Questions

- 当前无阻塞性开放问题。本 change 直接采用“活动根对象本地空间 + 平面过原点”的固定语义推进。
